# 学术论文写作指令

本目录中的 `.tex` 文件是中文学术论文，不是软件设计说明书、开发日志或用户手册。

## 一、事实来源

论文中的算法定义、参数、实验结果和数值结论必须以以下内容为事实依据：

1. `../src/` 中的当前实现；
2. `../tests/` 中经过验证的不变量和边界；
3. `../output/diagnostics/` 或 `evidence/` 中的实验结果；
4. 现有 `.tex` 中已经建立且未被新证据推翻的数学定义。

不得根据配置名称或函数名称推测实验结论。
不得虚构不存在的结果、统计显著性、最优性证明或工业结论。

## 二、论文叙事要求

写作时应采用学术论文的逻辑：

研究问题
→ 数学建模
→ 方法设计
→ 实验设置
→ 实验结果
→ 原因分析
→ 局限性与结论

不要按照代码文件、类、函数、CLI 命令或开发步骤组织正文。

除“实现与复现说明”小节外，正文应避免出现：

- 文件名；
- Python 类名或函数名；
- CLI 参数；
- YAML 字段；
- 输出目录；
- “新增、支持、实现、生成、兼容、重构”等开发报告式措辞。

## 三、表达转换

应将工程表述转换为学术表述。

不写：

“程序新增 adaptive restart，并支持 max_attempts=80。”

应写：

“为降低搜索结果对初始顺序的敏感性，本文采用具有最大尝试次数约束的自适应重启策略。”

不写：

“程序输出 restart_records.jsonl。”

应写：

“实验完整记录每次重启的目标值、Offset 分配及运行时间，以支持可复现性和解多样性分析。”

不写：

“CP-SAT 命令用于验证 GCLS。”

应写：

“本文构建带稳态峰值预算约束的 CP-SAT 模型，用于判断启发式解与预算内最优平方和之间的差距。”

## 四、语气与术语

使用客观、克制、可验证的中文学术表达。

优先使用：

- “本文提出……”
- “实验结果表明……”
- “在给定离散模型与参数条件下……”
- “该结果说明……”
- “一种可能的解释是……”
- “尚不能据此证明……”
- “求得高质量近似解”

避免使用：

- “效果非常好”
- “完美解决”
- “显然最优”
- “牛逼”
- “彻底证明”
- “工业上一定适用”
- “求得最优 Offset”

除 CP-SAT 返回 OPTIMAL 或穷举验证外，不得使用“全局最优”。

## 五、结果写作规则

每个实验结果段落应包含：

1. 实验目的；
2. 控制变量和评价指标；
3. 最重要的数值结果；
4. 对结果的解释；
5. 结论边界。

不要只罗列表格数值。
不要把“测试通过”写成算法有效性的主要证据。
不要把 Offset assignment 不同直接解释为目标不稳定；应区分目标稳定性与配置多样性。

## 六、LaTeX 编辑规则

- 保留现有章节结构、标签、引用和公式编号，除非任务明确要求调整；
- 新增图表时使用 `\label` 和 `\ref`；
- 数学符号必须与全文现有定义一致；
- 百分比、单位和变量格式保持统一；
- 不在正文中粘贴长代码或 JSON；
- 不修改与当前实验无关的章节；
- 修改后检查未定义引用、重复标签、公式符号冲突和表格宽度；
- 先提交修改计划，再编辑 `.tex`；
- 完成后列出修改章节、采用的证据文件及仍未写入论文的结果。
